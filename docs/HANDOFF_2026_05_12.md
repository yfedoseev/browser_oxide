## What landed (in commit order)

### `6ef419b` — stealth: Function.prototype.toString patch + WebIDL [[Construct]] hygiene

1. **Re-enabled `Function.prototype.toString` patch** in `stealth_bootstrap.js:11-38`.
   Was disabled May 9. Without it, sensors that call
   `Function.prototype.toString.call(maskedFn)` bypass per-instance toString
   overrides and read the raw JS source of every wrapped Web API method.
   Verified via new `check_tostring_test`:
   `Function.prototype.toString.call(navigator.plugins.item)` →
   `"function item() { [native code] }"` (previously: raw source).

2. **WebIDL methods are no longer constructors**. `_defProtoMethod` wraps the
   impl in an object-literal method-shorthand (which lacks `[[Construct]]`)
   AND explicitly preserves `fn.length` for reflection parity with Chrome
   WebIDL arity. ~50 inline shorthand wrappers in `window_bootstrap.js` use
   the same pattern.

### `4de702e` — dom: iterator-protocol fix + DOMTokenList iterability

1. Iterator objects returned from collection `[Symbol.iterator]()` now expose
   their own `[Symbol.iterator]() { return this; }` so they can themselves be
   iterated. Spec-compliant.

2. **`DOMTokenList` (element.classList) is now iterable**: added
   `[Symbol.iterator]`, `entries/keys/values/forEach/item`. Real Chrome
   DOMTokenList yields each token string on iteration. Before this commit,
   `[...element.classList]` threw "non-iterable".

### `0b050ec` — test: 5 chrome_compat diagnostics

- `check_iterators_test` — iterator self-iterability
- `check_ctors_test` — `new fn()` throws for WebIDL methods
- `check_spread_test` — spread succeeds on DOM iterables
- `check_tostring_test` — `Function.prototype.toString.call(fn)` returns `[native code]`
- `check_ao_candidates_test` — exhaustive Kasada `ao`-target probe (showed DOMTokenList was the only missing iterable)

Plus `kasada_symbol_iterator_probe` (ignored, network) — installs an
`Object.prototype[Symbol.iterator]` getter that logs Kasada-frame
fall-through accesses. Currently captures only 1 access during a full
canadagoose load (`Array.from({length:N})` from ips.js — doesn't throw).

### `db32af9` — test: kasada_capture_ips_js

Hooks `HTMLScriptElement.src` setter to capture the per-session ips.js URL,

## What didn't work — and why

### The `ao` spread probe stayed flagged

Attacked the W4a doc's `ao` field hypothesis: "Kasada spreads something
non-iterable in our engine." Built `check_ao_candidates_test` to walk every
plausible spread target (plugins, mimeTypes, fonts, classList, FormData,
URLSearchParams, RTCRtpReceiver.getCapabilities, etc.).

Found: `DOMTokenList` was the only candidate that lacked Symbol.iterator.
Added it. Re-ran `kasada_error_blob_capture` — still 6 e=1 flags including
`ao`. The fix is correct (verified, spec-compliant) but doesn't crack
this specific probe.

Static analysis of the captured ips.js source (526 KB) revealed:
- **0 `...` spread occurrences** in the source — babel-transpiled
- **0 `for...of` loops**, **0 `Promise.all`**, **0 `Reflect.apply`**
- **0 `arguments`**, **0 `.apply`** — even though these are essential to
  any real JS code, the entire surface is encoded as `Function`-constructed
  VM bytecode
- regenerator-runtime visible at offset 515783 (handles async/generator
  iteration; not directly involved in `ao`)
- 2 Array.from calls (both safe: `Array.from(int.toString(), Number)` and
  `Array.from({length: 169}, fn)`)
- **The error string "Invalid attempt to spread non-iterable instance" is
  NOT present in plaintext** — neither in ips.js source nor in the
  captured VM opcode bodies

**Conclusion**: the probe receiver + error template live in Kasada's
XXTEA-encrypted const table (opcode #5 in the W4a doc). Identifying the
`ao` target requires breaking that cipher.

### THIN-BODY (cloudflare.com, prime-video) — not real failures

Both sites render correctly when loaded in isolation:
- `cloudflare.com`: 1,316,007-byte body, title "Cloudflare: Build for the agent era"
- `primevideo.com`: 367,629-byte body, title "Welcome to Prime Video"

But classified as THIN-BODY in the parallel sweep. Root cause: when 4
ParallelPager workers compete for CPU/network/V8 isolates, page navigation
captures the body snapshot before the SPA finishes hydrating. **This is a
test-measurement artifact, not an engine bug.** No JS fix would help.

This is also why the score moves ±2 between sweep runs — flaky timing.

## Next-session tooling — what to build first

These two diagnostic tools are the prerequisite for ANY meaningful Kasada
work. Without them, you're guessing at probe targets.

### Tool 1: XXTEA key capture

The Kasada VM cipher block (opcode #5, body length 1170 chars) takes its
key from the first argument array. Key is a `[p, u]` pair of 32-bit integers.

Plan:
- Hook `new Function(body)` to identify the cipher handler (body contains
  `var d=2654435769` — the XXTEA golden-ratio constant)
- Wrap the resulting function so the FIRST call's `arguments[0]` is logged
- Decrypt the const table offline via Python (analog to existing
  `decrypt_report.py` which uses XOR `omgtopkek` for transport)
- Map probe IDs (`ao`, `crs`, `csc`, etc.) to their receiver-fetching
  opcode sequences

Estimate: 1-2 days.

### Tool 2: VM dispatcher instrumentation

The U dispatcher at ips.js line 5 col=116 has the shape:
```
i = u[t[e.X[0]++]]; try { var N = i(e, G, F, s, n, r); ... }
```

Where `u` is the opcode table, `t[e.X[0]++]` is the next opcode index, and
`i(e, ...)` calls the handler. Hook this dispatcher to log:
- Opcode index
- Handler body (truncated)
- Return value (null=continue, else=branch)
- Any exception caught by `D(e, o)`

Cross-reference the failed-probe stacks (we have these from
`kasada_typeerror_stack_capture`) with the opcode log. The opcode that
executes right BEFORE the throw is the one with the missing receiver.

Estimate: 1-2 days (probably less since we have the dispatcher source).

### Tool 3 (auxiliary): full sweep variance characterization

Run parallel sweep 5 times in a row, average the L3 count. Should show
~115 ± 2 with the 2 timing-flake sites swapping in and out. Useful for
distinguishing real improvements from noise.

Estimate: 2 hours.

## What NOT to do next session

- Don't pile more "fix this Web API" patches without first identifying the
  actual probe target via Tool 1 or Tool 2. Every patch landed this session
  was correct and additive but didn't measurably move the bot-flag count
  because we were attacking shadows.
- Don't re-run the sequential sweep. The 18-site sequential→parallel gap
  is IP-velocity rate-limiting (Akamai detects request patterns), not
  engine quality. Parallel is the canonical measurement.
- Don't drop the toString patch again. It's the foundation other masking
  builds on.
