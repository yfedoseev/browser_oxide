# 04 — The `isTrusted` Authenticity Crux

> **Thesis.** Every behavioral signal browser_oxide synthesizes (mouse
> trajectories, keystroke dwell/flight, scroll, focus) is *worthless to a
> tier-2 anti-bot* if the events that carry it read `Event.isTrusted ===
> false`, **or** if `isTrusted` is implemented in a way that itself
> fingerprints us. browser_oxide is a from-scratch in-process engine that
> *owns* the DOM, the event loop, and the `Event` objects. Unlike a CDP
> tool (Playwright/Puppeteer/Patchright) — which can only flip `isTrusted`
> through `Input.dispatchMouseEvent` and is constrained by Chromium's C++
> input pipeline — **we can mint genuinely-trusted events at the native/op
> level and place `isTrusted` exactly where real Chrome puts it.** This is
> a *structural advantage* that no JS-injection or CDP-driven competitor
> has. This doc specifies how to exploit it.

## 1. What a real browser does

### 1.1 The two-class event model

In Chrome/Blink, **every `Event` object carries an internal `isTrusted`
flag** (Blink: `Event::is_trusted_`, set via `Event::SetTrusted`). The flag
is initialized by exactly one of two paths:

- **User-agent / OS path → `isTrusted = true`.** Physical input (mouse,
  keyboard, touchscreen, wheel) enters Blink through the C++ input pipeline
  (`WebInputEvent` → `EventHandler` → `EventDispatcher`). When Blink
  constructs the corresponding `MouseEvent`/`PointerEvent`/`KeyboardEvent`
  it sets `is_trusted_ = true`. Scripts **cannot** reach this constructor.
- **Script path → `isTrusted = false`.** `new MouseEvent(...)`,
  `document.createEvent(...)`, `dispatchEvent(...)` all produce events with
  `is_trusted_ = false`. The spec is explicit: *"`isTrusted` … is true when
  the event was generated by the user agent … and false when … dispatched
  via `EventTarget.dispatchEvent()`."* ([MDN][mdn-istrusted])

`isTrusted` is exposed to script as a **getter on `Event.prototype`** — it
is *not* an own data property of each event instance. This is verifiable:

```js
// Real Chrome 148:
Object.getOwnPropertyDescriptor(Event.prototype, 'isTrusted')
// → { get: ƒ isTrusted(), set: undefined, enumerable: true, configurable: true }
Object.prototype.hasOwnProperty.call(new Event('x'), 'isTrusted')
// → false   (the instance has NO own 'isTrusted'; it inherits the getter)
e.isTrusted = true            // silently ignored (no setter)
Object.getOwnPropertyDescriptor(new Event('x'), 'isTrusted')
// → undefined
```

The getter is **non-configurable in the sense that pages cannot
`defineProperty` over it on an individual instance to a *truthful*
value** — Blink reads the internal flag, not a JS slot.

### 1.2 What anti-bots actually check

Anti-bot client scripts are *event-driven*: they `addEventListener` on
`mousemove`/`pointermove`/`pointerrawupdate`/`wheel`/`keydown`/`keyup`/
`touchstart` (typically at `window`, capture phase) and accumulate the
biometric stream. The defenses then use `isTrusted` in three escalating
ways:

1. **Per-event gate.** The handler drops or down-weights any event with
   `e.isTrusted === false` *before* it enters the biometric buffer. This is
   the cheapest and most common check. If our synthetic stream reads false,
   the buffer the vendor scores is effectively empty → "zero human entropy"
   → block. ([Akamai sensor.js / bmak][voidmob], [warandcode][warandcode])
2. **Descriptor / shape probe.** Sophisticated scripts (PerimeterX/HUMAN,
   DataDome, parts of Akamai bmak) don't trust the *value* — they probe the
   *plumbing*: `Object.getOwnPropertyDescriptor(Event.prototype,
   'isTrusted')` must be an accessor; an instance must **not** own
   `isTrusted`; the getter's `toString()` must be `function isTrusted() {
   [native code] }`; setting `e.isTrusted = true` must be a no-op. A bot
   that flips the value with `Object.defineProperty(e, 'isTrusted',
   {value:true})` *creates an own data property*, which is itself a tell.
   ([medium/detecting-fake-events][medium-fake])
3. **Trusted-only sink.** The vendor only arms its "human present" flag
   from a `click`/`keydown` that satisfies `isTrusted` **and** is part of a
   coherent sequence (`pointerdown`→`pointerup`→`click` with monotonic
   timestamps and a preceding `pointermove` trail).

`isTrusted: true` produced by a *real* input pipeline is therefore not just
a boolean — it is a **promise of an entire coherent native event
sequence**, and detectors increasingly validate the sequence, not just the
flag. ([browser-use #3829: even when a tool forces `isTrusted:true`, the
*shape* of the synthetic sequence leaks][browseruse])

### 1.3 Coalesced events & `pointerrawupdate` (the high-frequency tell)

Real pointer input on Chrome is delivered coalesced: a single
`pointermove`/`mousemove` dispatched per animation frame carries the
intra-frame samples, retrievable via
`PointerEvent.getCoalescedEvents()` → a list of `isTrusted:true`
`PointerEvent`s at the raw device rate (~120–1000 Hz), each with its own
`timeStamp`. `pointerrawupdate` is the un-throttled, high-frequency event
for precision apps. Both are **secure-context-gated** (added by
[w3c/pointerevents#318][w3c318]) precisely *because* they expose
high-resolution input entropy — which means anti-bots use the *absence* or
*emptiness* of `getCoalescedEvents()` as a bot signal: a real human cursor
produces 2–8 coalesced sub-events per frame at speed; a synthetic
`dispatchEvent` produces a `pointermove` whose `getCoalescedEvents()`
returns `[event]` (length 1) or throws. ([MDN getCoalescedEvents][mdn-coal])

## 2. What browser_oxide does today (file:line)

### 2.1 `isTrusted` is an own data property set from a Symbol — and is overwritten per-instance

`crates/js_runtime/src/js/event_bootstrap.js`:

```js
 7   const _TRUSTED = Symbol.for('__bo_trusted__');
 9   class Event {
10       constructor(type, options = {}) {
...
19           this.isTrusted = !!(options && options[_TRUSTED] === true);   // ← OWN DATA PROPERTY
20           this.timeStamp = performance.now();
```

**Three distinct, independently-fatal problems on this one line:**

- **(P1) Wrong plumbing — own data property, not a prototype getter.**
  Real Chrome: `isTrusted` is a getter on `Event.prototype`, instances own
  nothing. browser_oxide: every event instance has an **own, writable,
  enumerable** `isTrusted` data property. The §1.2-2 descriptor probe
  trivially distinguishes us:
  - `Object.getOwnPropertyDescriptor(Event.prototype, 'isTrusted')` → in BO
    this is **`undefined`** (the prototype has no such property — it's on the
    instance). In Chrome it's an accessor. **Hard tell.**
  - `new Event('x').hasOwnProperty('isTrusted')` → BO returns **`true`**,
    Chrome returns **`false`**. **Hard tell.**
  - In BO `e.isTrusted = false` *works* (writable own prop). In Chrome it's
    a silent no-op (no setter). **Hard tell.**

- **(P2) The trust source is a JS-reachable Symbol.** The only way to get
  `isTrusted:true` today is to pass `{ [Symbol.for('__bo_trusted__')]:
  true }`. `Symbol.for` lives in the **global symbol registry**, so *any
  page script can mint the identical Symbol* and forge trusted events —
  `new Event('click', {[Symbol.for('__bo_trusted__')]: true}).isTrusted ===
  true`. This is asserted as a *feature* by the test
  `behavioral_polish.rs:108-116`, but from a detection standpoint it means
  (a) the trust gate is not actually protected, and (b) a vendor that
  *enumerates the global Symbol registry* (or simply tries the obvious
  names) can detect the hook. The comment at `event_bootstrap.js:3-6`
  claims "pages don't know this Symbol" — **false**, `Symbol.for` is global.

- **(P3) humanize re-stamps `isTrusted` with `Object.defineProperty`,
  creating the exact own-data-property tell that detectors hunt for.**
  `crates/browser/src/js/humanize.js`:
  ```js
  104  function _dispatch(target, event) {
  105      try { Object.defineProperty(event, 'isTrusted', { value: true, configurable: true }); }
  106      catch (e) {}
  107      target.dispatchEvent(event);
  108  }
  ...
  429      try { Object.defineProperty(mev, 'isTrusted', { value: true, configurable: true }); } catch (_) {}
  441      try { Object.defineProperty(pev, 'isTrusted', { value: true, configurable: true }); } catch (_) {}
  ```
  Every humanize-dispatched event gets an **own, non-enumerable,
  non-writable, configurable** `isTrusted` data property. So even if we
  fixed P1 by moving the getter to the prototype, humanize would *re-break
  it per event* by shadowing the prototype getter with an own value prop.
  Worse: `configurable:true` + `value:true` is a signature no real event
  ever has.

### 2.2 There is NO native event-dispatch path — everything is JS `dispatchEvent`

Confirmed by exhaustive grep: `crates/js_runtime/src/extensions/input_ext.rs`
exposes **only data generators**, never a dispatcher:

- `op_behavior_random` (`input_ext.rs:54`)
- `op_human_mouse_path` (`input_ext.rs:65`)
- `op_human_typing_delays` (`input_ext.rs:111`)
- `op_human_keystroke_schedule` (`input_ext.rs:155`)
- `op_behavior_mouse_trajectory` (`stealth_ext.rs:185`)

**No `op_dispatch_trusted_event`, no `op_input_*`, no native event sink
exists.** The comment in `event_bootstrap.js:3-6` ("Rust dispatchers
(input_ext.rs etc.) pass `{ [Symbol.for('__bo_trusted__')]: true }`") refers
to code **that does not exist** — no Rust caller ever constructs or
dispatches an event. 100% of input synthesis is JS: humanize.js builds
`new MouseEvent(...)` / `new PointerEvent(...)` / `new KeyboardEvent(...)`
and calls `target.dispatchEvent(...)` (humanize.js:104-108, 138, 147,
246-248, 265-267, 280-285, 430-444).

This is the central "engineering exists, wiring does not" pattern called out
in MEMORY: the Rust trajectory/keystroke *math* is excellent and wired into
JS, but the **event delivery is pure JS `dispatchEvent`** — so it inherits
all of §2.1's problems and can never be more trusted than P2 allows.

### 2.3 Dispatch path does not propagate `isTrusted` through retargeting, and re-uses one event object

`event_bootstrap.js:316-366` (`_dispatchEvent`) walks capture/target/bubble
phases correctly, but:

- It **reuses the same `event` object** across all three phases and across
  `window`+`document`+`body` (humanize.js dispatches the *same* `mev` to
  three targets, lines 430-432). Real Chrome dispatches one event through
  one tree walk; re-dispatching the identical object to three unrelated
  targets sets `event.target` three times and is observable (a listener on
  `window` that captured the object earlier sees `target` mutate). Minor,
  but a coherence tell.
- It does not implement `event` (the global `window.event`) population, nor
  `sourceCapabilities` / `UIEvent.sourceCapabilities.firesTouchEvents`
  (another trusted-input correlate some scripts read).

### 2.4 No coalesced / raw pointer support

Grep confirms `getCoalescedEvents`, `getPredictedEvents`,
`pointerrawupdate` (as a dispatched event) **do not exist**:

- `PointerEvent` class (`event_bootstrap.js:123-137`) has **no
  `getCoalescedEvents()` / `getPredictedEvents()` methods**.
- `onpointerrawupdate` appears only as an **on-handler name string** in the
  interface lists (`interfaces_bootstrap.js:138`,
  `dom_bootstrap.js:2712`) — no `pointerrawupdate` event is ever
  *dispatched*.

So a detector calling `pointermoveEvent.getCoalescedEvents()` gets a
`TypeError` (method missing) → instant bot classification on any site that
probes it. And our `pointermove` stream is one-event-per-move with no
sub-frame samples — the §1.3 emptiness tell.

|---|---|---|---|
| `isTrusted` location | getter on `Event.prototype` | own data prop on instance | **Yes (hard)** |
| `instance.hasOwnProperty('isTrusted')` | `false` | `true` | **Yes (hard)** |
| `getOwnPropertyDescriptor(Event.prototype,'isTrusted')` | accessor | `undefined` | **Yes (hard)** |
| `e.isTrusted = false` effect | no-op | mutates | **Yes (hard)** |

## 3. The gap + the exact engine change to close it

The fix has two layers: **(A) make `isTrusted` plumbing
indistinguishable from Chrome** (cheap, pure-JS, closes all the §2.1/§2.5
descriptor tells), and **(B) mint genuinely-trusted events at the op level**
so the trust source is *not* a forgeable JS Symbol and the events come from
"outside" page JS (the structural moat). Then **(C)** add coalesced/raw
pointer support.

### 3A. Plumbing: native getter on `Event.prototype` backed by a non-forgeable internal slot

**Goal:** `isTrusted` becomes a prototype accessor that reads a per-event
*internal* flag pages cannot reach, write, or even observe as an own
property.

**Mechanism (JS-side, no Rust required for the plumbing):**

1. In `event_bootstrap.js`, replace the constructor line 19. Store the trust
   bit in a **module-private `WeakMap`** keyed by the event instance — *not*
   an own property, *not* a global Symbol:

   ```js
   // module-private; never exported, never a global Symbol
   const _trustMap = new WeakMap();

   class Event {
       constructor(type, options = {}) {
           ...
           // default: untrusted. Trust is only granted by the native
           // mint path (3B), never by page-reachable options.
           _trustMap.set(this, false);
           ...
       }
   }
   ```

2. Define the getter **once on `Event.prototype`**, masked as native:

   ```js
   Object.defineProperty(Event.prototype, 'isTrusted', {
       get: function isTrusted() { return _trustMap.get(this) === true; },
       set: undefined,                 // no setter — matches Chrome
       enumerable: true,
       configurable: true,             // Chrome's descriptor IS configurable
   });
   if (typeof _maskFunction === 'function')
       _maskFunction(Object.getOwnPropertyDescriptor(Event.prototype,'isTrusted').get, 'get isTrusted');
   ```

   This single change makes **all four hard tells in §2.5 disappear**:
   instances own nothing, the descriptor is an accessor on the prototype,
   assignment is a silent no-op, and the getter `toString()`s as native.

3. **Delete the global Symbol grant entirely** (P2). Trust must come from a
   path page JS cannot invoke. The only internal grant is a module-private
   function the bootstrap *closes over* and hands to the native bridge:

   ```js
   // handed to the trusted-dispatch op bridge only; NOT on globalThis
   function _markTrusted(ev) { _trustMap.set(ev, true); }
   ```

**Effort:** ~30 LOC in `event_bootstrap.js`. **Confidence: high.** This is
pure plumbing and closes every descriptor-shape tell.

### 3B. The structural moat: an op that constructs + marks + dispatches trusted events

This is what a CDP tool **cannot** do as cleanly and what justifies the
"from-scratch" architecture. Today input is JS `dispatchEvent`; we replace
it with a native op so the event originates *outside* page script and is
marked trusted via the closed-over `_markTrusted` (3A.3) that page JS has no
handle to.

**Two viable designs (pick B1 for v0.1.0, evolve to B2):**

- **B1 — Native-bridge dispatch (recommended first step).** Add a single
  internal JS function, installed by `event_bootstrap.js` into a
  module-private slot (closed over `_markTrusted` + the real
  `EventTarget.prototype.dispatchEvent`), and expose a thin *op* that calls
  it. The op signature:

  ```rust
  // crates/js_runtime/src/extensions/input_ext.rs  (NEW)
  /// Dispatch a fully-trusted synthetic input event. The event is
  /// constructed and marked isTrusted=true via the bootstrap's
  /// module-private _markTrusted, so the trust bit is NOT reachable from
  /// page JS (unlike the removed Symbol.for grant). target_node is the
  /// arena NodeId (-999 = window, 0 = document).
  #[op2]
  pub fn op_dispatch_trusted_event(
      scope: &mut v8::HandleScope,
      #[smi] target_node: i32,
      #[string] kind: &str,        // "mousemove" | "pointermove" | "keydown" ...
      #[serde] init: EventInit,    // {clientX, clientY, key, code, ...}
  ) { /* call the closed-over native bridge in JS realm */ }
  ```

  The bridge JS (closed over `_trustMap`):

  ```js
  globalThis.__bo_internal_dispatch = function (targetNode, kind, init) {
      const ctor = _ctorFor(kind);          // MouseEvent/PointerEvent/...
      const ev = new ctor(kind, init);
      _markTrusted(ev);                      // <-- the only trust grant
      const target = _resolveNode(targetNode);
      _realDispatch.call(target, ev);        // captured EventTarget.proto.dispatchEvent
  };
  ```

  Crucially `__bo_internal_dispatch` should be **deleted from `globalThis`
  by `cleanup_bootstrap.js`** after the runtime captures a reference (same
  pattern the codebase already uses for `__akamai_events`/internal slots),
  so page JS can never call it.

  **Why this beats CDP:** Playwright/Patchright must round-trip through
  `Input.dispatchMouseEvent` (CDP), which (a) is itself detectable via the
  CDP runtime/`Runtime.enable` side-effects browser_oxide doesn't have (the
  no-CDP moat — see `docs/v0.1.0-frontier-workflows/*`), and (b) cannot
  attach `getCoalescedEvents` samples. We mint the event *in-realm* with the
  trust bit set, no external protocol, no CDP surface.

- **B2 — True native pipeline (later).** Route `op_dispatch_trusted_event`
  through `crates/event_loop` + `crates/dom`'s native dispatcher so the
  event is built in Rust and the JS `Event` wrapper is materialized lazily
  with the trust bit pre-set, exactly mirroring Blink's
  `WebInputEvent → EventDispatcher` path. This removes even the in-realm JS
  constructor call from the trust path. Heavier; defer until B1 ships.

**Wire humanize.js to the op.** Replace `_dispatch` (humanize.js:104-108)
and the inline `defineProperty` blocks (429, 441) with calls to the op:

```js
function _dispatch(targetNode, kind, init) {
    const ops = Deno?.core?.ops;
    if (ops?.op_dispatch_trusted_event) ops.op_dispatch_trusted_event(targetNode, kind, init);
    else { /* test-only fallback: plain new+dispatchEvent (untrusted, OK for unit tests) */ }
}
```

This deletes **all three `defineProperty(... 'isTrusted' ...)` sites** and
makes humanize events trusted via the §3A.3 closed-over grant — closing P3.

**Effort:** ~120–180 LOC (one op + JS bridge + humanize rewiring + node
resolution). **Confidence: high** for B1; the node-id resolution
(`_getNodeId`, used in `event_bootstrap.js:272`) already exists.

### 3C. Coalesced & raw pointer support

Add to `PointerEvent` (`event_bootstrap.js:123-137`):

```js
getCoalescedEvents() {
    // Stored by the native mint path; for a single synthetic move,
    // return a non-empty list of sub-frame PointerEvents (2-6) carrying
    // the intra-frame samples the trajectory generator already produces.
    return _coalescedMap.get(this) || [this];
}
getPredictedEvents() { return []; }   // empty is valid & common
```

And in B1's bridge, when dispatching a `pointermove`, also attach 2–6
coalesced sub-events (sample the §3 Rust trajectory between the previous and
current frame) and **dispatch a `pointerrawupdate`** immediately before each
`pointermove` (it must precede `pointermove`, be non-bubbling-to-`click`,
and read `isTrusted:true`). This converts the §2.4 hard tell (method throws)
into a *positive* high-frequency entropy source that real humans have and
most bots lack.

**Effort:** ~60 LOC. **Confidence: medium** (needs the coalesced sample
buffer threaded from the trajectory generator; the math already exists in
`stealth::behavior::mouse_trajectory`, 125 Hz samples — `input_ext.rs:62`).

### 3D. Sequence coherence (so the trusted flag survives §1.2-3 sink checks)

Once events are genuinely trusted, ensure the *sequence* is coherent:
`pointerdown` → `pointerup` → `click` with monotonic `timeStamp`, a
preceding `pointermove` trail, matching `clientX/Y` within a few px, and
`buttons` transitions (0 → 1 on down → 0 on up). humanize.js already fires
move trails; extend it to emit a coherent click sequence via the op. This is
the difference between "isTrusted passes the gate" and "isTrusted passes the
human-present sink." **Effort:** ~40 LOC. **Confidence: medium.**

## 4. Which behavioral/API-gated sites this unblocks

The `isTrusted` plumbing fixes (3A) are **API-correctness** fixes — they
remove a hard fingerprint that gates *any* site running a descriptor probe,
independent of behavior. The trusted-dispatch + coalesced fixes (3B/3C/3D)
are **behavioral-entropy** fixes that feed the vendor scorers.

| Site / vendor | Gated on | Which fix |
|---|---|---|
| **bestbuy** (Akamai BMP) | §3.3 of `SITE_bestbuy.md` names `isTrusted=false` as one of two known BMP weaknesses; field-65 mouse model + trusted gate | 3A + 3B + 3D (+ the Rust-generator trajectory gap, BEHAVIORAL §3.1 Gap #3/#4) |
| **yelp** (DataDome) | `_initialCoordsList` harvested from `mousemove`+`pointermove` at `window`; DataDome down-weights `isTrusted=false`; getCoalescedEvents emptiness | 3A + 3B + 3C |
| **Kasada** (canadagoose/hyatt/realtor) | sensor VM scores pointer entropy + descriptor integrity; the own-data-property `isTrusted` is exactly the kind of prototype-shape tell Kasada's VM enumerates | 3A (descriptor) + 3B |
| **PerimeterX / HUMAN** | known to run `getOwnPropertyDescriptor(Event.prototype,'isTrusted')` + native-`toString` probe + per-event gate | 3A (closes the descriptor tell outright) + 3B |
| **ozon / wildberries** | behavioral + (possibly) geo-IP; trusted entropy is necessary-not-sufficient — flag IP separately per mission brief | 3A + 3B, then re-measure |
| **Any site probing `pointermove.getCoalescedEvents()`** | method currently throws | 3C |

**Ranking of *what unblocks the most*:** 3A is the highest-leverage single
change because the descriptor tells are **IP-independent, behavior-
independent hard fingerprints** present on *every* page load — they fail us
before any behavior is even sampled. 3B/3D then convert "not detected as a
bot" into "scored as a present human."

> **Honesty caveat (cite `SITE_bestbuy.md` §3.3 / `VENDOR_akamai.md`):** in
> the *public* engine the empty solver set means nothing *posts*
> `__akamai_events` to Akamai — so for Akamai specifically, trusted events
> are a **prerequisite the private `vendor_solvers` consumes**, not a
> standalone flip. For DataDome/PerimeterX/Kasada (which score *in-page*),
> trusted events + correct descriptor shape *are* directly load-bearing.

assert_eq!(evaluate("typeof Object.getOwnPropertyDescriptor(Event.prototype,'isTrusted').get").await, "function");
assert_eq!(evaluate("let e=new Event('x'); e.isTrusted=true; e.isTrusted").await, "false"); // setter no-op
assert_eq!(evaluate("new Event('x', {[Symbol.for('__bo_trusted__')]: true}).isTrusted").await, "false"); // grant deleted
// after R4:
assert_eq!(evaluate("typeof new PointerEvent('pointermove').getCoalescedEvents").await, "function");
