## Session 2 update (2026-04-27 afternoon) — CI/CD fix

**All tests now pass.** `cargo test --workspace -- --test-threads=1` exits clean.
Full write-up: `docs/SESSION_2026_04_27_RESULTS.md`.

### What was broken

Three root-cause bugs, all in the Worker postMessage + bootstrap pipeline:

1. **`Object.freeze` on `__boxide`** (`dom_bootstrap.js` line 1791): the final
   `__boxide` object was installed as `Object.freeze({ _getNodeId })`. In
   non-strict mode, `structured_clone.js`'s attempt to add `serializeForWire`
   and `deserializeFromWire` to the frozen object silently failed. Every
   `Worker.postMessage(arrayBuffer)` fell back to raw JSON, stringifying
   ArrayBuffers as `{}`. **Fix**: removed `Object.freeze` — object is now
   mutable so `structured_clone.js` can add the wire functions.

2. **`globalThis.__boxide` used after cleanup** (`window_bootstrap.js` poll
   timer, `worker_bootstrap.js` postMessage): `cleanup_bootstrap.js` deletes
   `globalThis.__boxide`. Two call sites used the live global lookup instead of
   the closure-captured `_boxide` variable, so they got `undefined` at runtime.
   **Fix**: both call sites now use the captured `_boxide` closure variable.

3. **Worker `dispatchEvent` undefined**: `worker_bootstrap.js` called
   `self.dispatchEvent(event)` in its `drainOnce` poll loop, but the worker
   scope had no EventTarget. Every incoming message silently threw and was
   swallowed. **Fix**: added `addEventListener`, `removeEventListener`,
   `dispatchEvent` to the worker global scope.

4. **localStorage/sessionStorage lost on navigation**: `navigate_loop_internal`
   in `page.rs` discarded storage on `drop(page)`. **Fix**: capture
   `current_storage` before each `drop` and pass it to the next page build.

5. **`execute_script` arity**: callers in `workers/src/lib.rs` and
   `js_runtime/tests/` passed one argument to the two-argument signature.

### Key invariant (don't break this)

`_boxide` in `window_bootstrap.js` and `_boxide` in `worker_bootstrap.js` are
closure-captured object references. They survive `cleanup_bootstrap.js`'s
deletion of `globalThis.__boxide`. Never read `globalThis.__boxide` in runtime
code (poll timers, event handlers, postMessage) — always use the captured ref.

## 3. Architecture additions this session

### `crates/browser/src/page.rs`

**`V8DeadlineWatcher`** (RAII guard, lines 11-72): spawns a polling
`std::thread` (100 ms granularity) that calls
`v8::IsolateHandle::terminate_execution()` if the deadline elapses. On drop,
signals cancel + joins. Used at TWO phases:
- **Build-phase** (line ~1372): preempts CPU-bound inline-script chains
  (delta.com, taobao.com sync-fetch loops). Cancelled + reset before returning
  the page so V8 is usable for drain.
- **Nav-phase** (line ~752): preempts spin loops during `run_until_idle` drain.
  Min 5 s floor so even past-budget iterations have a safety net.

**Adaptive budget extension** (line ~786): after iter=0 build+drain, if no
challenge marker AND body > 20 KB, grant +25 s nav budget. Recovers heavy SPAs
(footlocker, walmart, udemy) without slowing CHL-page bail-outs.

**Bail-before-iter check** (line ~688): skip starting a new iteration if
`nav_t0.elapsed() >= nav_budget`. Returns the previously-built page rather
than producing a "Navigation loop terminated without returning a page" FAIL.

**MIN_RETRY_BUDGET guard** (lines 881, 1075): both cookie-delta retry path
AND pending-nav follow path check `remaining < 15 s` and return current page
instead of consuming it for a refetch that won't have time to complete.

### `crates/js_runtime/src/extensions/fetch_ext.rs`

**`MAX_SYNC_FETCH_PER_PAGE = 30`** (atomic counter): caps `op_net_fetch_sync`
calls per page. Reset by `Page::navigate_with_init` at each iteration start.
Bounds taobao mtop polling and delta DTM cascade.

### `crates/js_runtime/src/lib.rs`

**`isolate_handle()`** + **`cancel_terminate_execution()`** on
`BrowserJsRuntime`: exposes `v8::IsolateHandle::thread_safe_handle()` so the
deadline watcher can preempt V8 from another thread.

### `crates/net/src/cookies.rs`

**`save_to_file()` / `load_from_file()`** on `CookieJar`: serde-based JSON
persistence, atomic via tempfile + rename. Both `CookieJar` and `Cookie` now
derive `Serialize, Deserialize`.

### `crates/net/src/lib.rs`

**`HttpClient::new`** (line ~138): loads jar from `BOXIDE_COOKIE_JAR` if set.

**`store_set_cookies`** (line ~756): saves jar after every Set-Cookie when env
var is set.

**`x-kpsdk-ct` injection** (line ~554): injects cached session token alongside
`x-kpsdk-cd` (PoW) and `x-kpsdk-fc` (Flow 2). Required by Kasada strict-tier
per Hyper-Solutions docs.

### `crates/net/src/kasada_session.rs`

## 4. Critical findings (the non-obvious learnings)

### 4.1 Network-layer fingerprint is byte-identical to real Chrome 147

Verified via tls.peet.ws comparison (playwright Chrome vs our pipeline):

| Metric | Both sides |
|---|---|
| JA4 | `t13d1516h2_8daaf6152771_d8a2da3f94cd` |
| peetprint_hash | `1d4ffe9b0e34acac0bd883fa7f79d7b5` |
| akamai_fingerprint | `1:65536;2:0;4:6291456;6:262144\|15663105\|0\|m,a,s,p` |
| supported_groups | `X25519MLKEM768, X25519, P-256, P-384` |
| cipher_list | identical |

**Implication**: Our chrome_130 BoringSSL config (in `crates/net/src/tls.rs`)
already tracks current Chrome. Don't waste time bumping the TLS impersonation
to "chrome_147" — there's no such thing as a different config that would
matter.

### 4.2 Real Chrome's UA-reduction policy

Real Chrome 147 sends `User-Agent: ...Chrome/147.0.0.0 Safari/537.36`. The
`.0.0.0` is FROZEN — Chrome's UA-reduction policy since Chrome 110 (March
2023). The full version (`147.0.7727.117`) is ONLY exposed via
`sec-ch-ua-full-version-list` AFTER server advertises `Accept-CH`.

**We were sending the full version in UA — that's a 100% reliable bot signal**
that we now fixed (#78).

### 4.3 Real Chrome's client-hints policy

On a first-touch GET, real Chrome sends ONLY low-entropy hints
(`sec-ch-ua`, `sec-ch-ua-mobile`, `sec-ch-ua-platform`). The "high-entropy"
hints (`sec-ch-ua-arch`, `-bitness`, `-full-version-list`, `-model`,
`-platform-version`, `-wow64`) are sent ONLY after the server advertises them
via `Accept-CH` response header.

**We were sending all 6 high-entropy hints unconditionally on retries —
another bot signal** that we now fixed (#79).

We do NOT yet parse `Accept-CH` from server responses + dynamically opt-in.
That's a TODO if you want to handle Accept-CH-aware servers correctly.

### 4.4 Kasada uses MORE than cookies for trust

Verified empirically (#80, #81): even with `tkrm_alpekz_s1.3`,
`tkrm_alpekz_s1.3-ssn`, `AKA_A2` cookies persisted across runs (saved by run 1
to `BOXIDE_COOKIE_JAR`, loaded by run 2), Kasada strict-tier STILL returns
`x-kpsdk-r: 1-AA` (untrusted) and a fresh challenge. Real Chrome on the same
machine gets `x-kpsdk-r: 1-BwRHegk` (trusted) with no JS-VM execution
required.

**Conclusion**: Kasada keys reputation on a server-side cache built from
(IP, UA, TLS-handshake-history, JS-VM-execution-history, behavioral-entropy).
The first 4 we've matched. The last 2 require either:
- A real Chrome process attached to a real OS profile that's accumulated
  history over time
- Or a residential rotating proxy / Kasada-as-a-Service that maintains
  pre-warmed trusted identities

**This is unwinnable as a fingerprint-matching problem from a fresh Rust client.**

### 4.5 Our Kasada solver is provably correct

Captured via diagnostics: hyatt's /tl POST returns HTTP 200 + `x-kpsdk-cr: true`
+ `x-kpsdk-st` + `x-kpsdk-ct` (171–174 char session token). We:
- Capture and store all 3 tokens in `KasadaSessionStore` (`learn()`)
- Inject them on subsequent same-host requests (`cd_header`, `fc_header`,
  `ct_header`)
- 250 ms+jitter delay between PoW and retry (avoids 429 rate-limit, fixed
  in #76)

The protocol is correct. The server's reputation rejection is the gate, not
our solver.

### 4.6 Three V8 sites SKIP'd due to a real bug

`creepjs`, `bot.sannysoft.com`, `leboncoin.fr` all crash V8 with
`Builtins_InterpreterEntryTrampoline` infinite recursion regardless of
`RUST_MIN_STACK=16M`. Not a stack-size issue — it's a real recursive bug in
our globalThis prototype-walking shim. Fingerprint testers (creepjs,
sannysoft) probe `Object.getPrototypeOf` / `Reflect.ownKeys` exhaustively;
DataDome bundles (leboncoin) trigger the same path. Tracked as **task #60**.

### 4.7 CPU-bound V8 work cannot be preempted by `tokio::time::timeout`

A `tokio::time::timeout(60 s, future)` only fires at the future's await
points. If V8 is in a tight microtask loop or our `op_net_fetch_sync` is
blocking on `std::thread::join`, the future never yields → timeout never
fires. The fix is `v8::IsolateHandle::terminate_execution()` from a separate
thread (which is what `V8DeadlineWatcher` does). This was the breakthrough
that converted delta + taobao from TIMEOUT to L3 PASS.

### 4.8 `is_anti_bot_challenge()` is heuristic and over-triggers

The function checks for substrings like `bm_sz` (Akamai), `kpsdk` (Kasada),
`perimeterx`, etc. in the body. It returns true for nike.com (which contains
`bm_sz` in its real homepage HTML), triggering the cookie-delta retry path
that consumes the iter=0 page. Without our `MIN_RETRY_BUDGET` guard, this
caused `[FAIL]` on nike + walmart + canadagoose + hyatt. (Fixed #61.)

If you tighten `is_anti_bot_challenge`, also re-test the IP-bound CHL pages
(canadagoose 730 bytes, ozon 97 KB) to make sure they still correctly
trigger the post-settle retry path.

### 4.9 Sync-fetch deadlock pattern (from earlier session, still applies)

`op_net_fetch_sync` MUST build a fresh `HttpClient` rather than reuse the
shared `FETCH_CLIENT` — otherwise the spawned `tokio::Runtime::new_current_thread`
deadlocks because the pooled HTTP/2 connections' reader/writer tasks live
on the main runtime which is blocked waiting for the sync op to return.
See `crates/js_runtime/src/extensions/fetch_ext.rs` for the comment block.

If you add other sync ops that touch shared async state, follow the same
pattern.

## 6. Operational decisions needed

These are the **single biggest levers for further progress** and they're
NOT code work — they're ops/billing decisions:

### 6.1 Residential proxy?

If you spend $50–500/mo on a residential proxy (Bright Data / Oxylabs /
IPRoyal / Smartproxy) and route through stable identities, you'd likely
flip:
- **canadagoose, hyatt, adidas** (Kasada strict-tier) → L3
- **wildberries** (WBAAS IP-bound) → L3 (with Russian residential)
- **ozon** (in-house IP block) → L3 (with Russian residential)
- **possibly douyin** homepage captcha

That's **+5 sites for budget alone**, no code work.

### 6.2 Kasada-as-a-Service?

Hyper-Solutions or Thoughtful provide Kasada-bypass APIs that maintain
pre-warmed pools of trusted identities. Cost is per-request but quality is
guaranteed for the strict-tier targets. Worth considering if Kasada sites
are critical to the product.

### 6.3 Persistent browser profile?

Headless Chrome with a persistent profile dir (cookies + localStorage +
IndexedDB + ServiceWorker registrations) accumulates Kasada reputation
over weeks. This is more work than residential proxy but free.

We've SHIPPED the cookie persistence half via `BOXIDE_COOKIE_JAR`. Adding
localStorage/IndexedDB persistence is the natural next step — but only
worth doing if you've tried residential proxy first (since proxy alone
should fix the Kasada-strict bucket).

### 6.4 Don't bother with TLS impersonation work

We verified our chrome_130 BoringSSL config is byte-identical to Chrome 147
TLS handshake. **Don't waste time on chrome_147 TLS impersonation** — there's
no such thing that would change anything visible to servers. (Closed as
not-needed in #74.)

## 8. Don't repeat these mistakes

- **Don't `Object.freeze` bootstrap bridge objects** — `__boxide` is used as a
  shared mutable namespace by multiple bootstrap files. Freezing it prevents
  `structured_clone.js` from registering wire serializers, silently breaking
  `Worker.postMessage` for any non-primitive value. Non-strict-mode assignments
  to frozen objects produce no error, making this class of bug extremely hard
  to spot without explicit `console.log(typeof __boxide.serializeForWire)`.
- **Don't access `globalThis.__boxide` after bootstrap** — `cleanup_bootstrap.js`
  deletes it. All post-bootstrap code (Worker poll timer, postMessage, event
  handlers) must use the closure-captured `_boxide` variable, not a live
  `globalThis.__boxide` lookup.
- **Don't add new properties to runtime objects you'll freeze** — if any future
  bootstrap object needs to be shared across files, keep it mutable or expose
  only through function closures.
- **Don't try to "bump TLS impersonation to chrome_147"** — verified
  not-needed (#74). Our chrome_130 BoringSSL config IS byte-identical to
  Chrome 147 at the JA4/Akamai-FP level. Real Chrome's TLS handshake
  varies per-connection due to GREASE + extension permutation, and JA4
  hashes those out. Don't waste days here.
- **Don't trust the "is_anti_bot_challenge" heuristic** to be tight — it
  over-triggers on real homepages that contain vendor markers (nike has
  `bm_sz`, hyatt has `kpsdk` strings in the init script). Always pair with
  the budget guards.
- **Don't add high-entropy CH headers thinking they help** — they're a bot
  signal when sent without an Accept-CH advertisement. Real Chrome is
  conservative about what it advertises.
- **Don't believe IP-bound conclusions without playwright verification** —
  We initially concluded canadagoose/hyatt/adidas were IP-bound. Playwright
  proved it wrong. Always run a real Chromium against the same site/IP
  before reaching for "it's the IP".
- **Don't use `tokio::time::timeout` to bound CPU-bound work** — it doesn't
  preempt. Use `V8DeadlineWatcher` (or its template) for V8/sync-op work.
- **Don't reuse the shared FETCH_CLIENT in sync ops** — see §4.9 deadlock
  pattern. Build a fresh client.
