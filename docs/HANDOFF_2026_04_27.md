## Session 3 update (2026-04-27 evening) — WBAAS investigation + Kasada sync XHR

### What was shipped

1. **`op_net_xhr_sync`** (`fetch_ext.rs` line ~327): New synchronous XHR op for
   Kasada KPSDK's `/tl` POST. KPSDK calls `xhr.open('POST','/tl',false);
   xhr.send()` (sync) and reads `xhr.status` immediately — the async `fetch()`
   path cannot satisfy this because V8 is busy with PoW. The op spawns an OS
   thread with a fresh tokio runtime + fresh HttpClient (avoids H2 deadlock),
   transfers cookies both ways (FETCH_CLIENT → fresh → back), has a 15 s
   timeout. Wired into the XHR `send()` path in `window_bootstrap.js`.

2. **`op_fetch` 30-second timeout** (`fetch_ext.rs`): `tokio::time::timeout(30s)`
   around the HTTP call. Prevents Kasada's `/tl` endpoint black-holing
   (server never responds when the PoW solution is invalid).

3. **Extended drain timeout** (`page.rs` line ~778): was hardcoded `8s`, now
   `remaining_nav_budget.max(8s)`. After Kasada PoW takes 34 s the old 8 s
   drain expired immediately; this change lets post-PoW async ops (like the
   `/tl` response landing cookies) complete before the page is returned.

4. **XHR logging in page.rs init** (`page.rs` line ~1665): patched
   `XMLHttpRequest.prototype.open/send` so XHR requests appear in `__fetchLog`
   alongside `fetch()` requests. Exposed during the WBAAS investigation where
   sync XHR was suspected as the token-fetch mechanism (turned out WBAAS uses
   `fetch()`, not XHR — but the wrapper is useful for future SDK debugging).

5. **`wbaas_full_network_trace` diagnostic test** (`chrome_compat.rs` line
   ~4337): captures all `op_fetch` traffic + reads both `__fetchLog` and
   `__allRequests`. Used with `max_iterations=1` or `2` to avoid the tries-
   counter lockout (see §4.10).

### Debug logging — cleaned up (session 4, 2026-04-27)

The two diagnostic `eprintln!` blocks from the WBAAS investigation have been
removed from `fetch_ext.rs`:

- `op_fetch` wbaas/create-token/find-frontend URL pattern logger — removed
- `op_net_xhr_sync` per-call logger — removed

Remaining `eprintln!` calls in `fetch_ext.rs` are all legitimate operational

## Session 2 update (2026-04-27 afternoon) — CI/CD fix

**All tests now pass.** `cargo test --workspace -- --test-threads=1` exits clean.
Full write-up: `docs/SESSION_2026_04_27_RESULTS.md`.

### What was broken

Three root-cause bugs, all in the Worker postMessage + bootstrap pipeline:

1. **`Object.freeze` on `__boxide`** (`dom_bootstrap.js` line 1791): the final
   `__boxide` object was installed as `Object.freeze({ _getNodeId })`. In
   non-strict mode, `structured_clone.js`'s attempt to add `serializeForWire`
   and `deserializeFromWire` to the frozen object silently failed. Every
   `Worker.postMessage(arrayBuffer)` fell back to raw JSON, stringifying
   ArrayBuffers as `{}`. **Fix**: removed `Object.freeze` — object is now
   mutable so `structured_clone.js` can add the wire functions.

2. **`globalThis.__boxide` used after cleanup** (`window_bootstrap.js` poll
   timer, `worker_bootstrap.js` postMessage): `cleanup_bootstrap.js` deletes
   `globalThis.__boxide`. Two call sites used the live global lookup instead of
   the closure-captured `_boxide` variable, so they got `undefined` at runtime.
   **Fix**: both call sites now use the captured `_boxide` closure variable.

3. **Worker `dispatchEvent` undefined**: `worker_bootstrap.js` called
   `self.dispatchEvent(event)` in its `drainOnce` poll loop, but the worker
   scope had no EventTarget. Every incoming message silently threw and was
   swallowed. **Fix**: added `addEventListener`, `removeEventListener`,
   `dispatchEvent` to the worker global scope.

4. **localStorage/sessionStorage lost on navigation**: `navigate_loop_internal`
   in `page.rs` discarded storage on `drop(page)`. **Fix**: capture
   `current_storage` before each `drop` and pass it to the next page build.

5. **`execute_script` arity**: callers in `workers/src/lib.rs` and
   `js_runtime/tests/` passed one argument to the two-argument signature.

### Key invariant (don't break this)

`_boxide` in `window_bootstrap.js` and `_boxide` in `worker_bootstrap.js` are
closure-captured object references. They survive `cleanup_bootstrap.js`'s
deletion of `globalThis.__boxide`. Never read `globalThis.__boxide` in runtime
code (poll timers, event handlers, postMessage) — always use the captured ref.

## 3. Architecture additions

### `crates/browser/src/page.rs`

**`V8DeadlineWatcher`** (RAII guard, lines 11-72): spawns a polling
`std::thread` (100 ms granularity) that calls
`v8::IsolateHandle::terminate_execution()` if the deadline elapses. On drop,
signals cancel + joins. Used at TWO phases:
- **Build-phase** (line ~1372): preempts CPU-bound inline-script chains
  (delta.com, taobao.com sync-fetch loops). Cancelled + reset before returning
  the page so V8 is usable for drain.
- **Nav-phase** (line ~752): preempts spin loops during `run_until_idle` drain.
  Min 5 s floor so even past-budget iterations have a safety net.

**Adaptive budget extension** (line ~786): after iter=0 build+drain, if no
challenge marker AND body > 20 KB, grant +25 s nav budget. Recovers heavy SPAs
(footlocker, walmart, udemy) without slowing CHL-page bail-outs.

**Bail-before-iter check** (line ~688): skip starting a new iteration if
`nav_t0.elapsed() >= nav_budget`. Returns the previously-built page rather
than producing a "Navigation loop terminated without returning a page" FAIL.

**MIN_RETRY_BUDGET guard** (lines 881, 1075): both cookie-delta retry path
AND pending-nav follow path check `remaining < 15 s` and return current page
instead of consuming it for a refetch that won't have time to complete.

**XHR logging in init script** (line ~1665): `XMLHttpRequest.prototype.open`
and `send` are patched so XHR requests log to `window.__fetchLog` alongside
`fetch()` entries. Important for debugging SDKs that use XHR instead of fetch.
Added after WBAAS investigation (WBAAS turned out to use fetch(), but the
wrapper is useful for others).

**Extended drain timeout** (line ~778): was `8s`, now
`remaining_nav_budget.max(8s)`. After Kasada PoW takes 34 s the old 8 s drain
expired immediately; this lets post-PoW async fetch (like `/tl` response
cookies) settle before returning the page.

### `crates/js_runtime/src/extensions/fetch_ext.rs`

**`MAX_SYNC_FETCH_PER_PAGE = 30`** (atomic counter): caps `op_net_fetch_sync`
calls per page. Reset by `Page::navigate_with_init` at each iteration start.
Bounds taobao mtop polling and delta DTM cascade.

**`op_net_xhr_sync`** (line ~327): synchronous XHR for Kasada KPSDK `/tl`.
KPSDK does `xhr.open('POST', '/tl', false); xhr.send()` sync and reads
`xhr.status` immediately — impossible with async `fetch()` while PoW is
spinning. Implementation:
- Spawns OS thread with fresh `tokio::Runtime::new_current_thread` + fresh
  `HttpClient` (avoids H2 deadlock with shared FETCH_CLIENT — see §4.9).
- Copies cookies from `FETCH_CLIENT` jar into fresh client before request.
- Copies response `Set-Cookie` headers back into `FETCH_CLIENT` jar after.
- 15 s timeout. Returns JSON `{status, url, headers: [[k,v],...], body}`.
- Wired in `window_bootstrap.js` XHR `send()` for `async=false` path only.

**`op_fetch` 30-second timeout**: `tokio::time::timeout(30s)` wraps the HTTP
call. Kasada's `/tl` endpoint silently black-holes requests when the PoW
solution is invalid — without this, the async op held the V8 event loop open
indefinitely.

### `crates/js_runtime/src/lib.rs`

**`isolate_handle()`** + **`cancel_terminate_execution()`** on
`BrowserJsRuntime`: exposes `v8::IsolateHandle::thread_safe_handle()` so the
deadline watcher can preempt V8 from another thread.

### `crates/net/src/cookies.rs`

**`save_to_file()` / `load_from_file()`** on `CookieJar`: serde-based JSON
persistence, atomic via tempfile + rename. Both `CookieJar` and `Cookie` now
derive `Serialize, Deserialize`.

### `crates/net/src/lib.rs`

**`store_set_cookies`** (line ~756): saves jar after every Set-Cookie when env

**`x-kpsdk-ct` injection** (line ~554): injects cached session token alongside

## 4. Critical findings (the non-obvious learnings)

### 4.1 Network-layer fingerprint is byte-identical to real Chrome 147

Verified via tls.peet.ws comparison (playwright Chrome vs our pipeline):

| Metric | Both sides |
|---|---|
| JA4 | `t13d1516h2_8daaf6152771_d8a2da3f94cd` |
| peetprint_hash | `1d4ffe9b0e34acac0bd883fa7f79d7b5` |
| akamai_fingerprint | `1:65536;2:0;4:6291456;6:262144\|15663105\|0\|m,a,s,p` |
| supported_groups | `X25519MLKEM768, X25519, P-256, P-384` |

**Implication**: Our chrome_130 BoringSSL config already tracks current Chrome.
Don't waste time bumping TLS impersonation to "chrome_147".

### 4.2 Real Chrome's UA-reduction policy

Real Chrome 147 sends `User-Agent: ...Chrome/147.0.0.0 Safari/537.36`. The
`.0.0.0` is FROZEN since Chrome 110 (March 2023). The full version is ONLY
in `sec-ch-ua-full-version-list` after `Accept-CH`.

### 4.3 Real Chrome's client-hints policy

First-touch GET: only low-entropy hints (`sec-ch-ua`, `-mobile`, `-platform`).
High-entropy hints only after server sends `Accept-CH`. Sending unconditionally
is a bot signal.

### 4.4 Kasada uses MORE than cookies for trust

Even with cookies persisted across runs, Kasada strict-tier STILL returns
`x-kpsdk-r: 1-AA` (untrusted). Server-side reputation keyed on (IP, UA,
TLS-handshake-history, JS-VM-execution-history, behavioral-entropy). The first
4 we've matched. The last 2 require real Chrome profile history or residential
proxy. **This is unwinnable as a fingerprint-matching problem from a fresh Rust client.**

### 4.5 Our Kasada solver is provably correct

Hyatt's `/tl` POST returns HTTP 200 + `x-kpsdk-cr: true` + `x-kpsdk-st` +
`x-kpsdk-ct` (171–174 char session token). We capture and inject all 3.
The server's reputation rejection is the gate, not our solver.

### 4.6 Three V8 sites SKIP'd due to a real bug

`creepjs`, `bot.sannysoft.com`, `leboncoin.fr` all crash V8 with
`Builtins_InterpreterEntryTrampoline` infinite recursion. Not a stack-size
issue — a recursive bug in our globalThis prototype-walking shim. Tracked
as **task #60**.

### 4.7 CPU-bound V8 work cannot be preempted by `tokio::time::timeout`

`tokio::time::timeout` only fires at `await` points. If V8 is in a tight
microtask loop, the future never yields. Fix: `V8DeadlineWatcher` calls
`terminate_execution()` from a separate thread.

### 4.8 `is_anti_bot_challenge()` is heuristic and over-triggers

Checks substrings like `bm_sz`, `kpsdk` in body — returns true for nike.com
(real homepage). Without `MIN_RETRY_BUDGET` guard, causes `[FAIL]` on pages
that contain vendor markers in legitimate content.

### 4.9 Sync-fetch deadlock pattern

`op_net_fetch_sync` and `op_net_xhr_sync` MUST build fresh `HttpClient` rather
than reuse `FETCH_CLIENT`. The spawned `tokio::Runtime::new_current_thread`
would deadlock because the pooled HTTP/2 reader/writer tasks live on the main
runtime which is blocked waiting for the sync op to return. Fresh client, own
connection pool, no deadlock.

### 4.10 WBAAS full internals (Session 3 — definitive analysis)

**The fingerprint is NOT the problem.** Full trace via `op_fetch` logging:

```
iter=0:
  POST /find-frontend-settings          → 200 {"solverPath": "challenge_solver_v1.0.4.js", ...}
  POST /create-token {}                 → 498 {"challenge": {"scriptPath": "challenge_fingerprint_v1.0.23.js", "payload": "..."}}
  POST /create-token {solution 13 KB}   → 200 {"secureToken": "1.1000.hash.b64.sig"}  ← SCORE 1000/1000
iter=1:
  (same, same result — score 1000)
iter=2:
  (no /find-frontend-settings, no /create-token)
  POST /report 825b                     → 200  ← "Get token tries exceeded"
```

**SDK internals** (from `index-DQJ0L4Mq.js`):
- `post()` method uses `fetch()`, NOT `XMLHttpRequest`.
- Challenge-response flow: first `create-token {}` → 498 with a challenge
  payload → `challengeSolver.solve(challenge)` → retry with solution → 200.
- On success: `document.cookie = "x_wbaas_token=<secureToken>; path=/; secure;
  SameSite=None; max-age=1209600"` then `document.location.reload()`.
- Our `document.cookie` setter DOES sync to FETCH_CLIENT via `op_cookie_set`.

**The tries counter** is the surface-level blocker:
- `localStorage['x_wbaas_token_treshold']` (typo in SDK) stores
  `{tries: N, lastTryTime: timestamp}`.
- Default = `{tries: 1}`. Increments per full token-creation attempt.
- After `tries >= 3` within 60 s: throws "Get token tries exceeded" without
  making any API calls.
- `max_iterations=3` causes tries to accumulate to 3 by iteration 3 →
  iteration 3 sends just the error `/report` with no `create-token` attempts.
- **Do not diagnose WBAAS with `max_iterations > 2`** — you'll only see the
  error report, not the actual token-creation flow.

**Unanswered question**: after score-1000 token is set + `location.reload()`,
why does wildberries STILL serve a challenge page in the next iteration? This
is the actual root cause. Candidates:
1. **IP-level block** — datacenter IP rejected regardless of token (most likely)
2. **Cookie transport bug** — `x_wbaas_token` set with `secure; SameSite=None`
   but not included in the next GET (domain/path mismatch in FETCH_CLIENT jar)
3. **Race condition** — `location.reload()` nav fires before `op_cookie_set`
   completes (async)

**Next diagnostic step**: Capture the cookie header sent in the iter=1 GET
(after token set in iter=0). If `x_wbaas_token` is absent, it's cookie
transport bug. If present, it's IP-level.

**Confirmed wrong theories** (don't re-investigate):
- Canvas/WebGL fingerprint bad → WRONG, score is 1000/1000
- `create-token` uses XHR not fetch → WRONG, it uses `fetch()`
- Empty request body → WRONG, iter=0 sends 13 KB challenge solution
- The SDK failing to load scripts → WRONG, `challenge_fingerprint_v1.0.23.js`
  (126 KB) and `challenge_vm_fp_v1.1.0.js` (197 KB) are loaded and executed

## 8. Don't repeat these mistakes

- **Don't `Object.freeze` bootstrap bridge objects** — silently breaks
  `Worker.postMessage` for any non-primitive. No error in non-strict mode.
- **Don't access `globalThis.__boxide` after bootstrap** — deleted by cleanup.
  Use closure-captured `_boxide`.
- **Don't bump TLS to chrome_147** — verified not-needed (#74).
- **Don't trust `is_anti_bot_challenge()` tightly** — over-triggers (nike has
  `bm_sz`, hyatt has `kpsdk`). Always pair with MIN_RETRY_BUDGET guard.
- **Don't add high-entropy CH headers unconditionally** — bot signal without
  Accept-CH advertisement.
- **Don't conclude IP-bound without playwright verification** — run real
  Chromium on same site/IP first.
- **Don't use `tokio::time::timeout` to bound CPU-bound V8** — use
  V8DeadlineWatcher.
- **Don't reuse shared FETCH_CLIENT in sync ops** — deadlock. Build fresh
  client (§4.9).
- **Don't assume WBAAS fails on fingerprint scoring** — confirmed score=1000.
  The block is IP-level on the page reload after the challenge, not inside the
  challenge itself. Don't waste time on canvas/WebGL fingerprint fixes for
  wildberries.
- **Don't run WBAAS diagnostics with `max_iterations > 2`** — tries counter
  hits 3 by iteration 3, SDK skips all API calls and just sends error report.
  Gives completely misleading picture. Use `max_iterations=1` for diagnostics.
- **Don't investigate why `__fetchLog` misses WBAAS create-token calls** — it
  doesn't. The SDK captures `fetch` at module eval time; in some iteration
  orderings that's before page.rs's `__fetchLog` wrapper. Add temporary
  `eprintln!` to `op_fetch` in `fetch_ext.rs` to see all traffic
  authoritatively (remove after investigation).
