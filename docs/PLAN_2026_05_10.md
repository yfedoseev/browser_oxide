### W2. iphey URL parser bug (1 site → 1 site OPEN, ~30 minutes)
**Tiny one-line bug.**

Root cause: `crates/net/src/lib.rs:1595::resolve_redirect()` returns
`location.to_string()` verbatim for non-`/`-prefixed redirect targets:
```rust
} else {
    Ok(location.to_string())  // BUG: doesn't resolve against base
}
```
A redirect like `Location: index.html` or `Location: ../foo` gets passed
to `Url::parse()` downstream → no host → "no host in URL".

Fix: replace the entire function body with the standard URL-resolution
algorithm (`Url::join`):
```rust
fn resolve_redirect(current_url: &str, location: &str) -> Result<String, NetError> {
    let base = Url::parse(current_url).map_err(|e| NetError::Request(e.to_string()))?;
    let resolved = base.join(location).map_err(|e| NetError::Request(e.to_string()))?;
    Ok(resolved.to_string())
}
```
This handles absolute, root-relative, and relative-with-no-leading-slash
correctly per RFC 3986 §5.2.

Test: write a test in net module that exercises `resolve_redirect("https://a.com/x/y", "z.html")` → `https://a.com/x/z.html`. Then re-run iphey.

**Risk**: low. `Url::join` is the canonical implementation.

### W3. TLS ClientHello fingerprint (1+ sites: wildberries, possibly canadagoose H2 downgrade, possibly yandex passport, ~1-3 days)
**Biggest cross-cutting impact. Detailed research:** `docs/RESEARCH_TLS_FINGERPRINT_FIX_2026_05_10.md` (output of running research agent).

- ALPS extension payload
- ECH retry-config payload

4. Fix the deltas in `crates/net/src/tls.rs::chrome_connector()`.
5. Re-run all three blocked sites + verify other sites don't regress.

**Risk**: medium. TLS fingerprinting is finicky — fixing one fingerprint detector can subtly break another. Test broadly post-fix.

### W5. SPA hydration timeouts (5 sites: twitter, x-com, hulu, yandex-ru, h-m, also khanacademy partially, ~1 week)
**Biggest engineering investment. Ceiling = +5 sites.**

Root cause: SPA shells (React/Vue/Svelte) need 1-5 MB main bundle to
hydrate; our V8 (deno_core 0.311 = V8 12.x) is slower than headed Chrome's
V8 + has higher per-Promise overhead in our runtime; hydration doesn't
complete before our 25-45 s deadline.

Two-tier approach:

**Tier A (cheap, partial win, ~1 day):**
- Bump per-host nav_budget for known SPA-shell sites in `crates/browser/src/page.rs::navigate_loop_internal()`:
  ```rust
  Some(h) if h.ends_with("twitter.com")
      || h.ends_with("x.com")
      || h.ends_with("hulu.com")
      || h.ends_with("yandex.ru")
      || h.ends_with("hm.com")
      || h.ends_with("khanacademy.org") => 90_000,  // 90s for SPA hydration
  ```
- Add an early-exit signal: when `document.querySelector('#root, #app, #__next, [data-reactroot]')?.children.length > 0` AND body > 50KB, return immediately even if event loop hasn't reached idle.
- Expected: maybe 2-3 of the 5 sites pass.

**Tier B (deeper, ~1 week):**
- Profile a single SPA (twitter cleanest). Instrument `event_loop::run_until_idle` to log:
  - Microtask drain count per tick
  - Per-tick wall time
  - Pending-task type breakdown (timers vs Promises vs IO)
- Find quadratic patterns. Common culprits in deno_core-based engines:
  - `Promise.then` chains with many microtasks per tick
  - Missing `requestIdleCallback` (our impl falls back to setTimeout(fn, 0)?)
  - MutationObserver flooding — each DOM op fires N observers
  - IntersectionObserver fires at the wrong cadence
- Fix the worst offender; remeasure.

**Risk**: medium. Tier A is safe. Tier B touches the event loop core.

## Cross-cutting concerns

1. **Don't regress what passes.** Add `holistic_sweep_parallel` to nightly CI so any change that drops the pass count below baseline is caught immediately. Currently it's `#[ignore]` and only run manually.
2. **Capture-driven debugging works.** The `kasada_error_blob_capture` + `decrypt_report.py` pattern (this session) was the highest-leverage tool we built. Replicate for DataDome (W6) and Cloudflare (W7).
3. **Real Chrome reference captures.** Several workstreams (W3 TLS, W6 DataDome, W7 Cloudflare) need real Chrome traffic captures from the same machine + IP. Stand up a Playwright MCP setup and a tcpdump habit. One-time setup, days of value.
4. **The 19× memory advantage matters**. Per `docs/COMPARISON_2026_05_10.md`, oxide is 40MB vs Chrome's 750MB per page. For scraping at scale this is the value prop. Don't sacrifice it for marginal pass-rate gains — measure memory after each engine fix to ensure it stays sub-100MB.
5. **The 19% v8 perf gap (DOM creation: 13ms vs 1.9ms) deserves its own workstream eventually.** Not on the holistic-pass-rate critical path but matters for overall product quality.
