## What changed (uncommitted, organized by surface)

### 1. Sub-pixel layout — Blink LayoutUnit (1/64 px) quantization

**New file**: `crates/layout/src/layout_unit.rs` (114 lines).
**Why**: Akamai pHash and CreepJS rect-rounding probes hash `getBoundingClientRect()` floats. Real Chrome emits values that are exact multiples of `1/64` (e.g. `width: 1.296875` not `1.3`). Engines that pass raw f32/f64 fail the hash.
**Wiring**:
- `crates/layout/src/lib.rs` exports `LayoutUnit` and the `layout_unit` module.
- `crates/layout/src/query.rs::DOMRect::new` quantizes all 8 fields (`x, y, width, height, top, right, bottom, left`) at construction.
- `crates/layout/src/engine.rs::taffy_size`/`taffy_position` quantize Taffy's f32 layout output before exposing it to the rest of the engine.
**Test**: `crates/browser/tests/sub_pixel_layout.rs` (89 lines) — asserts `1.3px` div returns `1.296875`, exercises quantization on positions/sizes/transformed rects.

### 2. Canvas 2D path coverage

**Why**: The original `arc`, `arcTo`, `bezierCurveTo`, `quadraticCurveTo`, `closePath`, `setTransform`, `resetTransform`, `strokeText` were no-op stubs. Anti-bot vendors call these in their canvas fingerprints (Kasada `ips.js`, DataDome `tags.js`, CreepJS `paintCanvas`). No-ops produce a blank canvas → trivially detectable.
**Changes**:
- `crates/js_runtime/src/extensions/canvas_ext.rs`: 6 new `#[op2]` ops wiring to the Skia-backed `Canvas2D` impl in `crates/canvas/`.
- `crates/js_runtime/src/js/canvas_bootstrap.js`: ops wired into the JS `Path2D` / `CanvasRenderingContext2D` shims.
**Caveats** (review these before committing):
- `arcTo` is approximated as two `lineTo` calls. Real Blink expands `arcTo` to a bezier expansion of the SVG/Canvas spec arc. For most fingerprint scenes this is fine because Chrome produces a smoother curve so byte-equality fails anyway, but it is *not* spec-correct. Replace with byte-faithful Blink port if any vendor probe diffs the rasterized output.
- `ellipse` is approximated as `arc(x, y, max(rx, ry), …)`. Same caveat.
- `strokeText` is aliased to `fillText`. Different rasterization shape — fine for non-byte probes, wrong for shape comparators.
**Tests**:
- `crates/browser/tests/canvas_paths.rs` (191 lines) — execution-without-throw + non-blank-output assertions for each op.
- `crates/canvas/tests/png_chunks.rs` (37 lines) — verifies our PNG output uses Chrome's minimal chunk set: signature + IHDR + IDAT + IEND. No `pHYs` / `tIME` / `tEXt` leak.
- `chrome_compat.rs::canvas_drawing_produces_nonblank_data_url` — WBAAS-style canvas exercise + asserts data URL ≠ blank-canvas baseline.

### 3. Chrome surface trim — `chrome.runtime` and `chrome.webstore` removed

**File**: `crates/js_runtime/src/js/window_bootstrap.js`.
**Why**: Real Chrome 147 on a regular page exposes only `{app, csi, loadTimes}`. `chrome.runtime` is extension-only; `chrome.webstore` was removed in Chrome 126. The previous engine exposed both with stubbed listener handles — that's a documented bot signal for Kasada / Cloudflare BM / DataDome.
**Also updated**:
- `chrome.app` now exposes `getDetails / getIsInstalled / installState / runningState` matching Chrome 147 shape.
- `chrome.loadTimes()` flags (`connectionInfo`, `npnNegotiatedProtocol`, `wasFetchedViaSpdy`, `wasNpnNegotiated`, `wasAlternateProtocolAvailable`) are now conditional on `location.protocol`. HTTP/HTTPS pages report `"h2"`/`true`; about:blank/non-HTTP report `""`/`false`.
- `performance.memory.jsHeapSizeLimit` bumped from `2172649472` to `4294705152` (Chrome 147 reference).

### 4. Cross-realm iframe purity — mirror-realm constructors

**File**: `crates/js_runtime/src/js/dom_bootstrap.js` (around line 1745, ~140 new lines).
**Why**: Tier-1 vendors (DataDome, PerimeterX, Imperva, Cloudflare BM, CreepJS) probe `iframe.contentWindow.Navigator !== window.Navigator` and inspect `getOwnPropertyNames(proto)` lists. Real Chrome iframes are distinct realms; our engine previously aliased the parent realm's constructors so the identity check failed.
**Approach**: Parent-side mirror — `_buildRemoteRealm` constructs fresh function copies for ~40 constructors (`Navigator`, `Window`, `Document`, all common `HTML*Element`, plus the JS built-ins) with prototype-property-name lists matching the parent's. Each method/getter/setter is a fresh native-shape stub carrying `_nativeTag` so cross-realm `Function.prototype.toString.call(fn)` returns `"function NAME() { [native code] }"`.
**Limit**: This passes shape-and-identity probes. It does *not* pass cross-isolate execution probes (e.g. spinning microtasks inside the iframe realm and observing they don't leak into the parent). No vendor we know of currently does that.
**Test**: `crates/browser/tests/realm_purity.rs` (152 lines).

### 5. Element scroll state — real getters/setters

**File**: `crates/js_runtime/src/js/dom_bootstrap.js` (`Element` class).
**Why**: Previously `scrollTop = 0` (always) with empty setters. Behavioral telemetry (HUMAN/PerimeterX, Akamai sensor) reads back `scrollTop` after writing it; identical-zero across all elements is a flag.
**Now**: `_scrollState` `Map<nodeId, {top, left}>` backs `scrollTop`/`scrollLeft` getter/setter pairs. `scrollTo`, `scrollBy`, `scrollIntoView` implemented as spec-aligned no-rendering wrappers.

### 6. Yandex Metrika blocklist (controversial — review before committing)

**File**: `crates/js_runtime/src/js/dom_bootstrap.js` (script-loader path).
**What**: Hard-coded skip-list for `mc.yandex.ru/metrika/{tag,watch}.js` and `mc.yandex.ru/webvisor/`. When a site loads these as third-party scripts, we synthesize `load` events without executing them.
**Why**: Multiple sites including `bot.sannysoft.com` and `leboncoin.fr` load Yandex Metrika, which triggers the V8-shim recursion crash (issue #60) inside Yandex's own VM. Skipping it lets the rest of the page complete.
**Trade-off**: This is a **functional regression** — a real Chrome would execute these scripts. A vendor that diffs network traces (very rare) would catch this. The pragmatic case: Yandex Metrika contributes no fingerprint signal and crashing on it costs all subsequent tests on the page. Acceptable until #60 is fixed; remove afterward.

### 7. Synchronous XHR op for Kasada `/tl` POST (carryover from 04-27 session)

Already documented in `HANDOFF_2026_04_27.md` §"Session 3". Files changed today are limited to the XHR-to-`__fetchLog` wrapper in `page.rs:1665` — XHR requests now appear in the fetch log alongside `fetch()` requests.

### 8. `op_fetch` 30 s timeout

**File**: `crates/js_runtime/src/extensions/fetch_ext.rs::op_fetch`.
**Why**: Kasada's `/tl` endpoint black-holes requests with invalid PoW solutions (server never responds). Without a timeout, V8 holds the event loop open indefinitely. 30 s is the empirical Kasada upper bound for valid solutions.

### 9. Drain-timeout = `nav_budget` (not hardcoded 8 s)

**File**: `crates/browser/src/page.rs:769`.
**Why**: Kasada PoW takes 30+ seconds. The old hardcoded 8 s drain expired before the post-PoW `/tl` POST could land cookies. Now the drain uses `nav_budget.saturating_sub(elapsed).max(8s)`. The `V8DeadlineWatcher` (installed earlier in the navigate path) provides the hard kill, so analytics setInterval loops still get terminated.
