# Session Handoff — 2026-05-05 (Update 2)

## Summary of Work
This update completes the core implementation tasks for the Canada Goose (Kasada) bypass and Akamai A6 parity.

### 1. Robust Network Flow
- **Consolidated POST Logic**: Merged redundant POST methods into `post_bytes_with_exact_headers` in `crates/net/src/lib.rs`.
- **Kasada Injection**: The new method automatically injects `x-kpsdk-ct` and `x-kpsdk-fc` into all POSTs, including JS-initiated fetch/XHR calls.
- **Header Sanitization**: Implemented strict header filtering (removes `:host`, `connection`, etc.) to match real browser fetch behavior.
- **Body Dumping**: Restored `BOXIDE_DUMP_POST_DIR` support for auditing large payloads.
- **MFC Trigger**: Successfully wired `/tl` and `/r` POSTs to trigger a background `/mfc` fetch, ensuring tokens are updated as soon as a challenge is solved.

### 2. Behavioral Humanizer (Persistence)
- **Looping Humanizer**: Modified `crates/browser/src/js/humanize.js` to run a new cycle of mouse/scroll/focus events every 4 seconds.
- **Trust Accumulation**: This ensures that during the 12s Kasada PoW build phase, the engine is actively generating trusted input telemetry (Akamai `sensor_data` and Kasada internal logs).

### 3. Akamai v13 / Pixel Parity
- **Battery API Enumerability**: Updated `BatteryManager` in `window_bootstrap.js` to make all properties enumerable. This ensures they are captured by Akamai's `for..in` probe (the `bt` field).
- **Permissions API Verification**: Cross-checked `navigator.permissions.query` against the Akamai `nap` 20-digit state table. Confirmed our current implementation correctly handles "prompt", "granted", and "enum-rejected" (TypeError) states to produce the expected 1/2/4 digits.

### 4. Diagnostics & Findings
- **Full Blob Capture**: Successfully captured full 67KB error blobs from `reporting.cdndex.io/error` (`kasada_error_0.b64`). These are confirmed to be encrypted/compressed.
- **H2 Detection Confirmation**: Re-enabled ALPS but confirmed Canada Goose still downgrades us to HTTP/1.1. This identifies the TLS ClientHello as a primary remaining detection vector.
- **304 Rejection**: Confirmed that `/mfc` returns `304 Not Modified` even with `cache-control: no-cache` and no conditional headers, indicating a reputation-based block (likely linked to the TLS fingerprint).
